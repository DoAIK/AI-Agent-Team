# TSUGIAI LP構成設計書

## 戦術分析報告

### 結論:
エンジニア向けダークテーマLP。「AIエージェントが引き継ぎの"惰性"を殺す」という挑発的コピーで技術的好奇心を刺激し、アーキテクチャ図とライブデモ的演出で「これは本物だ」と確信させる構成。全8セクション、スクロール駆動アニメーション。

### 根拠は3つ:
1. **ターゲットはエンジニア** ── 曖昧なマーケティング文句ではなく、技術アーキテクチャ・API設計・エージェント協調パターンを見せることが最大の訴求力になる
2. **プロダクトの差別化ポイントが明確** ── 3エージェント協調、トラップ質問、危険キーワード検出。これらは「なぜそう設計したか」を語れば技術記事としても成立する深さがある
3. **製造業×AI×音声対話の組み合わせが新しい** ── B2B SaaSのLPは退屈になりがちだが、Gemini Live APIの音声対話とマルチフェーズUXは視覚的に魅せる素材が豊富

### 代案を2つ:
1. **代案A: ストーリーテリング型** ── 「ある工場の引き継ぎ事故」から始めて課題→解決→技術の順で見せる。感情に訴える構成。リスク: エンジニアには回りくどく感じる可能性がある
2. **代案B: インタラクティブデモ中心型** ── LP自体にチェックアウトのミニデモを埋め込み、体験させる。リスク: 実装コストが跳ね上がる。シングルページHTML制約との相性が悪い

### 最悪ケースを1つ:
技術を前面に出しすぎて「で、何が解決できるの？」が伝わらない。対策: Hero直下にPain Pointセクションを置き、課題→解決の文脈を先に作る。

### 次の一手を1つ:
この設計書をdriverに渡し、HTML実装に入る。criticには構成レビューを依頼。

### タイポグラフィ
- 見出し: `Inter` または `Space Grotesk`（Google Fonts CDN）
- 本文: `Inter`
- コード: `JetBrains Mono` または `Source Code Pro`

### アニメーション方針
- CSS only（JS最小限）
- `@keyframes` + `IntersectionObserver`（スクロールトリガー用の最小JS）
- `transition-duration: 0.3s` 基本
- `animation-timing-function: cubic-bezier(0.4, 0, 0.2, 1)`
- パフォーマンス優先: `transform` と `opacity` のみアニメーション

### レスポンシブ方針
- Mobile First
- ブレークポイント: `640px`（sm）、`768px`（md）、`1024px`（lg）
- モバイルでは横スクロールセクションを縦スタックに変更
- コードブロックは横スクロール許容

## 比較表（推奨案の根拠）

| 観点 | 推奨案（技術訴求型） | 代案A（ストーリー型） | 代案B（デモ体験型） |
|------|---------------------|---------------------|---------------------|
| 概要 | アーキテクチャとコードで魅せる | 事故ストーリーから入る | LP内にミニデモを埋め込む |
| ターゲット適合 | 高（エンジニア直撃） | 中（感情訴求は好み分かれる） | 高（体験は最強）|
| 実装コスト | 低〜中（HTML+CSS主体） | 低（テキスト主体） | 高（デモ実装が必要） |
| 技術的印象 | 高（コードが見える） | 低（技術は後半に回る） | 高（動くものがある） |
| 制約適合 | 高（シングルHTML可能） | 高 | 低（シングルHTMLでは困難） |
| 成功確率 | 高 | 中 | 低 |

## トレードオフ分析
- **訴求力 vs 実装コスト**: 推奨案はコードスニペットとCSS演出で「技術感」を出せる。デモ体験型は最強だが、制約と合わない
- **感情 vs 論理**: エンジニアターゲットでは論理（アーキテクチャ、設計判断の根拠）が刺さる。ストーリー型は経営層向き

## 推奨案
- 推奨: 技術訴求型（本設計書の構成）
- 理由: ターゲット適合性、制約適合性、実装コストの3点で最もバランスが良い

## 失敗モード

| # | 失敗パターン | 発生条件 | 影響 | 対策 |
|---|------------|----------|------|------|
| 1 | 技術に寄りすぎて課題が伝わらない | Section 2が弱い場合 | 「で、何が嬉しいの？」 | Pain Pointを先に置き、文脈を作る |
| 2 | コードが多すぎて読む気を失う | Section 5,7が冗長な場合 | 離脱率上昇 | コードは最小限、コメントで意図を補足 |
| 3 | モバイルでレイアウト崩壊 | CSS Grid/Flexの設計不備 | モバイルユーザーの離脱 | Mobile First設計、ブレークポイント検証 |
| 4 | 読み込みが遅い | フォント・アニメーション過多 | FCP悪化 | フォントはdisplay:swap、アニメーションはCSS only |

## プラン切り替え条件
- 推奨案 → 代案A: エンジニア以外（経営層・現場管理者）もターゲットに含める必要が判明した場合
- 推奨案 → 代案B: シングルHTMLの制約が緩和され、バックエンド接続が許容される場合

## 撤退条件
- シングルHTMLの制約下でインタラクティブ要素が10個を超える場合 → 複数ページ構成への切り替えを検討
- Tailwind CDNの読み込みが3秒を超える環境が想定される場合 → インラインCSS onlyに切り替え
